# Mate — 코덱스 핸드오프 (Round 9)

> 이전 라운드: `Mate_Codex_Handoff_R1~R8.md`. 이번 라운드: 카카오/SMS 키가 아직 발급 전이라, 개발을 막지 않기 위한 임시 인증 우회안.
> ⚠️ 아래는 전부 **개발 전용**이다. 실제 서비스 오픈 전 반드시 진짜 카카오+SMS로 교체해야 한다 — PRD P1 "익명 불가" 원칙이 노쇼 이력 추적·신고 시 신원 확인 같은 안전장치의 전제이기 때문.

---

## 원칙: "수동 플래그"가 아니라 "키 존재 여부로 자동 분기"

별도의 `DEV_MODE=true` 같은 플래그를 만들지 않는다. 대신 **환경변수(카카오/SMS 키)가 없으면 자동으로 임시 경로를 타도록** 만든다 — 이렇게 하면 나중에 진짜 키를 채워 넣는 순간 자동으로 정상 경로로 전환되고, "프로덕션 배포 전에 플래그 끄는 걸 깜빡함" 같은 사고를 원천 차단한다.

---

## 카카오 로그인 → Supabase Email Magic Link로 대체

카카오 OAuth는 부분 우회가 불가능하다(앱 등록 자체가 있어야 시도라도 가능). `KAKAO_CLIENT_ID` 환경변수가 없으면:
- `/login` 화면에서 카카오 로그인 버튼 대신 이메일 입력 + "매직링크 받기" 버튼 표시
- `supabase.auth.signInWithOtp({ email })` 사용 (Supabase 프로젝트에 Email provider는 기본 활성화되어 있음, 별도 설정 불필요)
- 이후 온보딩·프로필 생성 로직은 카카오 로그인 때와 동일 — `auth.users.id`만 있으면 되므로 나머지 코드는 안 바뀜

## 휴대폰 인증 → OTP 플로우는 그대로 두고, "발송"만 우회

이건 완전히 건너뛰지 말고 **플로우 자체는 실제로 돌리는 걸 추천**한다 — 어차피 만들어둔 `phone_otp_requests` 테이블/rate limit 로직을 지금 검증해볼 수 있고, 나중에 SMS 벤더 키만 꽂으면 될 정도로 코드 변경을 최소화할 수 있다.

`POST /api/auth/phone/request-otp` 안에서:
```
if (!process.env.SMS_VENDOR_API_KEY) {
  // 실제 SMS 발송 대신, 생성된 OTP를 응답에 그대로 포함(개발 전용)
  return { data: { devOtp: otp }, error: null }
}
// 키가 있으면 원래대로 실제 SMS 벤더 API 호출
```
프론트는 `devOtp`가 응답에 있으면 화면에 살짝 보여주거나 자동으로 입력창에 채워서 테스트를 편하게 만들면 됨. `phone_verified=true` 처리, `attempt_count`/만료 로직은 전부 실제 코드 그대로 탄다.

---

## VAPID / Web Push는 이미 실제로 붙었음

VAPID 키는 준비 완료 상태 — 이 부분은 우회 없이 **지금 바로 실제 Web Push 구독·발송을 테스트**할 수 있다. 카카오/SMS와 달리 이건 임시 처리가 필요 없다.
