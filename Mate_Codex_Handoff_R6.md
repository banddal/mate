## 이번 라운드 = 버그 수정 (신규 기능 아님)

| 문제 | 프론트/운영에 영향 |
|---|---|
| 소멸 즉시성 버그 | 없음 — 서버 RLS만 수정, 프론트는 이미 "종료된 방은 안 보인다"고 가정하고 만들면 됨(원래 계획대로) |
| 유저 정지 미집행 | 없음 — Supabase Auth 표준 `banned_until` 사용, 프론트는 로그인 실패 시 "계정이 정지되었습니다" 에러 메시지 처리만 추가하면 됨 |
| 호스트 완료 이력 공백 | 없음 — 순수 백엔드 로직 |
| §7 문구 오류 | 없음 — 문서 오탈자 수준 |
| **프로필 마스킹 조건** | **있음** — 아래 참고 |
| 현금성 패턴 범위 | 없음 — 서버 검수 로직만 |
| 카드 검수 알림 누락 | **있음** — 아래 참고 |

## 프론트엔드 작업 (변경 필요한 것만)

| 화면 | 변경 내용 |
|---|---|
| `/cards/[id]/applicants` (호스트용) | 응답이 **두 계층**으로 옴: (1) 항상 오는 필드 — 신청사유·인증여부·완료이력·노쇼이력 (2) 신청자수 > 정원일 때만 오는 필드 — 블러 프로필·평점지수·성사지수(null이면 "경쟁 없음" UI로, 마스킹 UI 자체를 렌더링하지 말 것) |
| 로그인 실패 처리 | Supabase Auth가 정지된 계정의 로그인을 거부할 때의 에러 코드를 잡아서 "계정이 일시/영구 정지되었습니다" 안내 필요 |
| 알림함/푸시 | `notifications.type='card_review_resolved'` 케이스 추가 — "카드가 승인되어 피드에 올라갔어요" / "카드가 반려되었어요(사유는 일반화된 문구)" |

## 운영(Ops) 작업

- 신고 처리로 유저를 정지시킬 때 서버가 **Supabase Admin API**(`updateUserById`, `ban_duration`)를 호출한다는 점만 인지 — 별도 벤더/인프라 추가 없음, 기존 Supabase 프로젝트 권한으로 충분
