## 1. 라우트별 스펙

| 경로 | 화면 | 스펙 |
|---|---|---|
| `/login` | 로그인 | Google OAuth 중심. 이메일 매직링크/SMTP는 사용하지 않음 |
| `/onboarding` | 최초 1회 | 닉네임·연령대·성별·관심 카테고리. 휴대폰 인증은 §4 우회안 참조 |
| `/feed` | 카드 리스트 | 필터 chip(오늘/이번주/마감임박/카테고리), pull-to-refresh, **프로필 사진 노출 금지** |
| `/cards/new` | 카드 생성(멀티스텝) | 기본정보 → 호스트가 건 것(1급 필드) → 마감시간. **카테고리 선택지에 L3 자체를 넣지 말 것**(API가 어차피 차단하지만 UI에서부터 안 보이게) |
| `/cards/[id]` | 상세+신청 | bottom sheet 모달, 사유 100자(`shared/config.ts`의 `APPLICATION_REASON_MAX_LENGTH` 참조, 하드코딩 금지) |
| `/cards/[id]/applicants` | 호스트 전용 | §2 참조 (2계층 응답) |
| `/rooms/[id]` | Mate Room 채팅 | 신고 버튼 sticky, 연락처 교환 경고 배너, **종료된 방은 재접근 불가**(서버가 즉시 차단) |
| `/rooms/[id]/review` | 종료 후기 | 이행 체크 3토글 + 신고 + 이모지(이모지는 로컬 표시만, 서버 전송 안 함) |
| `/alerts` | 상황 템플릿 구독 | **"사람" 구독 옵션 자체를 만들지 않음** |
| `/admin/*` | 관리자 | **역할 구분 없이 단일 화면** (moderator/superadmin 분리 없음) |

## 2. 신청자 목록 응답 — 2계층 (경쟁 여부에 따라 다름)

`GET /api/cards/:id/applicants` 응답:
- **항상 노출**: 신청 사유, 인증 여부, 완료/노쇼 이력
- **신청자수 > 정원일 때만 추가 노출**: 블러 프로필 사진, 평점 지수, 미팅 성사 지수 — `null`이면 숫자 대신 **"신규" 배지** 표시(표본 부족). 정확한 %나 표본수는 노출하지 말고 완곡한 서술형 문구 권장("다녀온 모임에서 시간 잘 지킴")

---

## 3. 카드 생성 응답 3가지 케이스

1. 성공 + 즉시 공개(`status='open'`) → 카드 상세로 이동
2. `status='pending_review'` → "검수 중입니다" 안내(호스트 본인만 확인 가능, 남에겐 안 보임)
3. `error.code='CONTENT_BLOCKED'` 또는 `L3_NOT_AVAILABLE_YET'` → 일반화된 안내 문구만("활동 중심으로 구체적으로 작성해주세요") — **정확히 어떤 단어가 걸렸는지 알려주는 UI 금지**(필터 회피 방지)

---

## 4. 초기 인증 우회안

키 유무로 **자동 분기**(수동 flag 아님) — 나중에 진짜 키 넣으면 자동으로 정상 경로로 전환됨.

- **로그인** → `/login`에서 Google OAuth 사용. SMTP 이슈로 이메일 매직링크/OTP는 비활성화
- **SMS** (`SMS_VENDOR_API_KEY` 없음) → OTP 플로우·DB 로직은 실제로 그대로 돌리되, `POST /api/auth/phone/request-otp`가 실제 SMS 대신 `devOtp`를 응답에 포함 → 프론트는 이 값이 오면 화면에 노출/자동입력해서 테스트

⚠️ 이건 개발 전용입니다. 실제 서비스 오픈 전 반드시 진짜 키로 교체(PRD P1 "익명 불가" 원칙의 전제).

## 5. 데이터 접근 규칙

- **카드 생성/상태변경(승인·거절·마감), 신청 승인, 신고 처리, 카드 검수, admin 관리 — 전부 API(Route Handler)를 통해서만.** Supabase 클라이언트로 직접 write하지 말 것 (RLS가 애초에 이걸 막아두기도 함 — `cards`/`applications`엔 client insert/update 정책이 없음)
- `profiles`(본인), `subscriptions`, `push_subscriptions`은 RLS로 직접 CRUD 허용되니 API 안 거쳐도 됨
- Mate Room 채팅(`messages`)은 Supabase Realtime으로 클라이언트가 직접 구독/전송 — RLS가 `rooms.status='active'`+참여자 여부를 실시간으로 강제
- DB 타입은 손으로 만들지 말고 `npx supabase gen types typescript --project-id <ID> > types/database.types.ts`로 생성

## 6. 절대 규칙 (통합, 지금 유효한 것만)

1. 카드 리스트/상세에 프로필 사진 노출 금지 (P1)
2. 감정 상태·취약성 공시 카드(실연/우울/외로움 등) 통과시키는 UI 금지 (P2)
3. 호스트가 걸 수 있는 건 상황에 묶인 비용뿐 — 현금 사례비 입력 필드 금지 (P3)
4. 신청 거절은 개별 통지 없음, 마감 시 일괄 "매칭이 확정되었어요" 메시지만
5. 모임 종료 후 채팅방 재접근·상대 프로필 재조회·팔로우/DM 기능 금지 (P4)
6. 후기 화면에 외모·매력·호감·인기 점수 필드 금지 — 사실 체크 4개 + 완료 배지만 (P5)
7. "호스트 구독"(사람 팔로우) 기능 금지
8. 카드 반려/차단 사유에 정확한 매칭 단어·패턴 노출 금지
9. `pending_review`/`rejected` 카드는 호스트 본인 외 누구에게도 어떤 경로로도 노출 금지
10. `subscription_match` 알림에 상대방 식별 정보 포함 금지
11. `admin_users` 부여/회수는 API를 통해서만 — "관리자 되기" 셀프서비스 플로우 금지
12. `escalated`/`suspended_perm` 신고 건은 어떤 화면에서도 "재검토"·"되돌리기" 액션 노출 금지
13. 평점 지수·미팅 성사 지수는 완곡한 서술형으로만 — 정확한 산출식·표본수 노출 금지
